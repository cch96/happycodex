## Design saturation and frozen release sequence

Fresh baseline-only challenger `/root/release_boundary_challenger_b` inspected only
baseline `48769821`, repository policy, supported engine/marketplace interfaces, and
the prior 0.3 delivery pattern. It did not read this plan, current worktree, post-
baseline commits, Root context, prior reviews, or agent output, and ran no test,
network, model, install, Fable, Goal, ref, or publication action. It returned
`NOT-YET` for the current baseline and seven decision-changing obligations. Root
independently reproduced the union. The earlier task
`/root/release_boundary_challenger` was interrupted without a terminal receipt; none
of its work is evidence.

Material alternatives:

1. **Version before live and certify once.** Rejected. It would bind behavior evidence
   directly to the final artifact, but collapses the repository's explicit boundary
   that kept every shipped 0.3 byte stable throughout the certification-engine phase.
   Release metadata belongs to a later artifact-only transition with its own gates.
2. **Live-certify unchanged 0.3 shipped bytes, then perform a zero-live 0.4 artifact
   transition.** Selected. First persist exact authority in the current ledger and
   source commit; run only the bound corpus/holdout scope; persist strictly later
   sanitized evidence; review the exact synthetic certified product; and commit the
   byte-identical reviewed ledger. Only then version the release and create a second
   `refresh_required` source whose prior evidence is the strictly older certified
   0.4 ledger. No live authority is allowed in that zero-call transition.
3. **Use one public install as both certification evidence and release proof.**
   Rejected in favor of two isolated installs. One install would temporarily expose a
   public branch whose ledger is still `refresh_required`. A local isolated
   marketplace install instead closes the artifact-only `offline_summary` gate before
   publication; after final certification and review, a separate fresh public Git
   marketplace install/invocation verifies the exact publishable branch.

The frozen release sequence is therefore:

1. After the canonical response, persist exact live authority in `current.json` and
   commit the authority-bearing source before any model call.
2. Run corpus and adaptive holdout through the sole official CLI into external raw
   directories. Stop on the first contract-defined failure. Copy only sanitized
   summaries/receipts into content-addressed evidence paths and commit them strictly
   after the source.
3. Construct the proposed certified ledger in an isolated synthetic commit, validate
   it, run one fresh exact-product review over that immutable product tree, require
   `GO`, then apply and commit byte-identical reviewed ledger bytes on the release
   branch. This preserves exact review identity while keeping review external.
4. Change only `.codex-plugin/plugin.json` to base `0.4.0` and use the official
   plugin-creator default cachebuster helper for one
   `0.4.0+codex.<UTC timestamp>` suffix. Add an accurate 0.4 change note to
   `README.md` and make only the minimal historical release-status correction in
   `AGENTS.md`. Skill, marketplace manifest, evaluator, runner, cases, oracles, and
   tests remain byte-identical.
5. Persist the artifact-only refresh source with prior evidence pointing to the
   strictly older certified ledger. Recompute impact: it must have zero live calls,
   unchanged semantic identity, null authority, and only the expected install/review
   gates; anything else stops.
6. In a fresh local isolated `HOME`/`CODEX_HOME`, install the exact committed source
   through a local marketplace, verify version plus source/installed Skill and package
   identities, and persist the strictly post-source `offline_summary`. Build the
   second certified ledger in a synthetic commit, run the final exact-product review,
   require `GO`, then commit the byte-identical reviewed ledger.
7. Push the retained release branch normally. In a second fresh isolated home, add
   `cch96/happycodex` at that exact release ref, install once, verify the resolved
   public commit/version/source-install equality, and perform only the minimum
   credential-safe invocation proof. No repository evidence changes after this exact
   product proof except this ExecPlan.
8. Snapshot the active 0.3 source/cache/list receipt, retain both the old cache and an
   exact source backup, mechanically replace the personal source with the reviewed
   0.4 package, and reinstall using `codex plugin add happycodex@personal`. Verify in
   new plugin state; on failure restore the 0.3 source/cache and verify rollback before
   stopping.
9. Open a PR to `main`, merge without squash/rebase so every source/evidence commit
   referenced by the ledger remains an ancestor, keep the release branch, tag the
   exact merged commit `v0.4.0`, publish the GitHub Release, and independently read
   back main/tag/release identities. Force, direct overwrite, and branch deletion
   remain forbidden.

Design saturation is reached: the selected two-transition/two-install flow is the
smallest sequence satisfying phase preservation, strict evidence ancestry, exact
review identity, public proof, rollback, and final GitHub delivery simultaneously.
